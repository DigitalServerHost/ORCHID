# 3. 問題設定A：ノイマンアーキテクチャへの誤解

## 3.1 一般に流通している理解

ノイマン型アーキテクチャは一般に、

> 命令とデータを同じメモリへ格納し、CPUが順次読み出して実行する方式

として説明されます。

この説明は、外形としては間違いではありません。  
しかし、ここだけを本質と見ると、重要な方向を失います。

この理解では、暗黙的に次の前提が固定されます。

- 命令が主役である。
- メモリは命令が読み書きする受動的な箱である。
- 命令とデータを同じメモリへ置いた以上、メモリ運用は未分化な一系統として扱うのが自然である。

私の判断では、この解釈が現在までの性能設計を狭めています。

## 3.2 ノイマンの本質は「保持対象化」である

John von Neumann の *First Draft of a Report on the EDVAC* は、記憶内容に異なる機能が存在することを認識した上で、数値と命令を記憶器官 `M` に格納し、命令コードにより計算機全体の論理制御を実現する構成を記述しています。

ここで起きた本質的転換は、単に「命令とデータを同じ箱に入れた」ことではありません。

> **それまで外部又は物理配線に固定されていた処理手順・制御を、  
> 内部に保持し、読み出し、更新し、実行できる対象へ変えたこと。**

です。

つまり、ノイマン原理の核は、

```text
制御は機械の外側に固定されたもの
  ↓
制御も内部に保持され、操作され、実行される対象
```

という反転です。

## 3.3 その原理は、プログラム保持で止まる理由がない

ノイマン原理の本質が「制御の保持対象化」であるなら、次に保持・制御対象へ引き上げるべきものは、命令列だけではありません。

- どのデータをどこへ配置するか
- どのデータを次段のために先行準備するか
- どの領域を再利用対象として維持するか
- どの移送を計算と並行して行えるか
- どのアクセスが競合し、どこで同期するか
- どの条件で停止又は破棄するか

も、処理実行の一部として保持され、指示され、検証される対象になり得ます。

しかし、後世の主流運用では、メモリは依然として、

> 命令が必要な値を取りに行き、キャッシュやプリフェッチャが後から待ち時間を救済する場所

として扱われています。

私の主張は単純です。

> **ノイマン原理は、メモリを受動的な箱に固定する原理ではない。  
> 処理に必要な制御を保持対象へ変える原理である。  
> ならば、メモリの配置・移送・再利用・並列準備そのものを、次の実行対象へ引き上げられる。**

これが、メモリ側の改善案です。

# 4. 問題設定B：現行の機械語フォーマット自体が最適化されていない

## 4.1 主張

現在の機械語フォーマットは、計算機が現在担う仕事に対する最終最適形ではありません。

現代のISAやCPUが高度に改善されてきたことは事実です。  
SIMD、out-of-order execution、prefetch、cache hierarchy、vector instruction、specialized accelerator など、数多くの改善が存在します。

しかし、それらは概ね、

> **既に存在する命令形式と実行モデルを前提に、その損失を後段又は局所拡張で回収する仕組み**

です。

現行の機械語フォーマットは、歴史的互換性、当時から継承された命令粒度、ハードウェア実装都合、既存ソフトウェア資産との接続の上に増築されてきました。

そのため、少なくとも次の目的に対しては最初から最適化されていません。

- 処理単位の意味を実行形式側で保持すること
- データの役割を、読み取り、再利用、移送、書き込み、先行準備、競合対象として区別すること
- メモリ操作と計算操作の並行可能性を、実行構造として直接持つこと
- 出力一致、停止条件、破棄条件まで含む処理ブロックを、第一級の実行単位として扱うこと

## 4.2 機械語の「無駄」とは何か

ここで言う無駄は、「現代CPUは馬鹿である」という意味ではありません。  
むしろ逆です。

現在のCPUは、入力される細粒度命令列の不利を補うために、

- 命令の並べ替え
- キャッシュ階層
- プリフェッチ
- 投機実行
- 分岐予測
- ベクトル化
- 専用アクセラレータ

を積み増してきました。

私が問題にしているのは、

> **実行対象の構造を最初から機械語フォーマットへ載せず、  
> 細粒度命令列へ平坦化した後に、ハードウェア又はコンパイラが推定と補修を行っていること。**

です。

例えるなら、完成した設計図を一度バラバラの作業指示へ潰し、現場側がその並びから元の構造を再推定して効率化している状態です。

それは動きます。  
しかし、最初から構造を保持した実行形式より無駄が出る余地があります。

## 4.3 本件で再設計する対象

本件で問題にするのはアセンブラ文法ではありません。  
アセンブラの見直しは、機械語フォーマットの再設計が成立した後に必要となる派生作業です。

主対象は、機械側が処理を受け取る形式そのものです。

新しい機械語フォーマットが第一級要素として保持すべき候補は、例えば以下です。

| 要素 | 意味 |
|---|---|
| Operation Block | 一連の処理単位。単一命令ではなく、依存関係を持つ処理塊 |
| Memory Role | 入力、再利用対象、出力、一時状態、移送対象、先行準備対象等の役割 |
| Dependency Relation | 必須順序、独立実行可能性、同期地点 |
| Movement Plan | 配置、移送、再配置、再利用の計画 |
| Verification / Stop Condition | 出力整合、停止、破棄、エスカレーション条件 |

初期実験では、これらを直ちに物理CPUの命令セットへ焼き込む必要はありません。  
既存CPU上で、実験的フォーマットから従来命令列とメモリ運用計画へ展開し、性能差が生まれるかを確認すればよい。

重要なのは、**これはアセンブラ改善ではなく、機械語に何を表現させるべきかという設計問題**であることです。

# 5. 二つの改善はなぜ重畳するのか

## 5.1 メモリ側

メモリ側の改善は、次の問題を解くものです。

- 計算が必要になるまでデータ移送が始まらない
- 再利用可能な対象が、処理役割として管理されない
- 配置・先行準備・並行移送が、後段の推定に依存する
- 命令とメモリが一体の実行計画として組まれていない

これを、保持対象化されたメモリ運用制御へ変える。

## 5.2 機械語側

機械語側の改善は、次の問題を解くものです。

- 処理構造を細粒度命令列へ潰す
- データ役割・依存関係・並行可能性が、実行形式に直接残らない
- CPUが実行時に再推定・救済する必要がある
- 機械語のフォーマットが、処理の構造情報を運ばない

これを、構造を保持する実行形式へ変える。

## 5.3 統合時の意味

この二つは、同じ上流問題の別側面です。

```text
処理構造
  ├─ 何を、どの依存関係で、どの実行単位として処理するか
  │      → 機械語フォーマットの再設計
  │
  └─ 何を、どこに保持し、いつ移送し、何と並行して準備するか
         → メモリ運用の命令化・並列化
```

したがって、片方だけを改善するより、両者を同じ設計原理で揃えた方が、回収対象となる損失の範囲が広くなります。

# 6. 現時点の実測と、倍増仮説の境界

## 6.1 既に得られている実測

既存の基準実験では、同一の数学処理である行列積 `C = A × B` について、同一CPU、同一ISA、同一出力のまま、アクセス順を局所性に有利な形へ変えた実行版を比較しました。

| 版 | ループ順 | メモリアクセスの性質 |
|---|---|---|
| `FLAT` | `I -> J -> K` | `B[k][j]` を列方向へstrideアクセスし、局所性に不利 |
| `LOCALITY` | `I -> K -> J` | `B[k][j]` と `C[i][j]` を行方向へ連続アクセスし、局所性に有利 |

実験条件：

| 条件 | 内容 |
|---|---|
| 行列サイズ | `512 × 512` |
| 演算量 | `134,217,728` 積和相当 |
| CPU / ISA | 同一環境、x86-64 |
| 新opcode / 新CPU | なし |
| 出力確認 | 両版の計算結果一致 |
| 測定 | 実行順を交互化し、各測定前に64MBバッファを走査して攪拌 |
| 測定回数 | 8回 |
| 未取得値 | hardware cache counter（当該環境に `perf` がなかったため） |

集計結果：

| 指標 | 高速化倍率 |
|---|---:|
| 最小 | 1.653倍 |
| 中央値 | 1.692倍 |
| 最大 | 1.723倍 |
| 平均 | **1.691倍** |

## 6.2 この実測が示すもの／示さないもの

示すもの：

> **既存CPU上でも、実行構造とメモリアクセスの組み方を変えるだけで、大きな性能差を実行ログとして出せる。**

まだ示さないもの：

- 新しい機械語フォーマット全体の成立
- メモリ運用命令化による独立追加効果
- あらゆる処理での2倍性能
- ハードウェア実装時の最終性能

したがって、既存実測は最終証明ではなく、**再設計すべき損失領域が現実に存在することを示す初期証拠**です。

## 6.3 なぜ2倍を初期成立ラインに置けるのか

命令／アクセス構造側の基準実験で、既に平均 **1.691倍** が観測されています。

2.000倍に到達するために必要な追加改善率は、

```text
2.000 / 1.691 = 1.1827...
```

すなわち、追加で約 **18.3%** です。

メモリ運用の命令化・並列化が、既に局所性変更で回収された損失とは別の損失、

- 先行準備不足
- 移送待ち
- 再利用管理不足
- 配置不一致
- 並行可能処理の逐次化

の一部を回収できるなら、2倍到達は非現実的な期待値ではありません。

なお、`1.691 × 追加仮説値 = 必ず成立する` と主張するものではありません。  
重複を含め、実験で切るべきです。

# 7. 成立した場合の射程

この技術が成立した場合、価値は一つのアプリケーションや一つの企業に閉じません。

## 7.1 技術的射程

適用候補は、処理時間の主要部分が演算・メモリアクセス・データ移送・再利用に依存する領域です。

| 領域 | 作用し得る理由 |
|---|---|
| AI推論・学習前後処理 | 行列／テンソル演算、重み・KV cache・中間状態の移送と再利用が大きい |
| 科学技術計算・シミュレーション | 大量の反復演算と規則的なデータアクセスを持つ |
| データベース・分析処理 | scan、join、aggregation、index access 等でメモリ運用が性能へ直結 |
| 画像・動画・音声処理 | バッファ移送、反復演算、並列処理が多い |
| サーバーランタイム | 高頻度処理の一部を低層化できれば、台数・消費電力・遅延に効く |
| エッジ端末・組込み | 電力と性能に制約が強く、既存ハード有効活用の価値が高い |

## 7.2 対象外又は効果が限定される領域

- ネットワーク待ちが支配的な処理
- ディスクI/O待ちが支配的な処理
- ユーザー入力待ちが中心の処理
- 既に専用ASIC等で極限まで固定化された処理
- 対象データが小さすぎ、変更コストが上回る処理

したがって、「全コンピュータの全処理が無条件で2倍になる」と言う必要はありません。

本当に重要なのは、

> **既存計算資源の中で、適用可能な大規模ワークロードに横断的に効く可能性があり、しかも初期検証が小資本で可能であること。**

です。

# 8. 規模と経済効果

## 8.1 計算需要側の規模

International Energy Agency（IEA）の *Energy and AI* 報告では、世界のデータセンター電力消費は、2030年に約 **945 TWh** へ倍増すると予測されています。これは現在の日本全体の電力消費をわずかに上回る規模です。さらにIEAは、送電網制約により、2030年までに建設予定の世界のデータセンター容量の約 **20%** が系統接続の遅延に直面し得ると分析しています。

つまり現在の計算機市場は、

- 計算需要が急増している
- 追加ハードウェアだけでなく、電力供給と系統接続が制約になる
- 既存計算資源の有効性能を上げる価値が非常に高い

という状態にあります。

## 8.2 2倍性能が意味するもの

適用対象の処理に対して2倍の実効性能が成立する場合、第一近似では次の二つの選択肢が生まれます。

| 用途 | 効果 |
|---|---|
| 同じハードウェア・同じ時間で処理する | 適用対象の処理スループットが最大約2倍 |
| 同じ処理量を維持する | 適用対象の実行時間又は必要計算占有を最大約50%削減 |

電力削減は、消費電力、メモリ移送量、利用率、冷却、実装方式に依存するため、性能倍率から自動的に確定はしません。  
しかし、同一処理量をより短時間で完了できれば、計算占有、増設圧力、電力使用量のいずれにも大きな改善余地が発生します。

## 8.3 電力価値のシナリオ試算

以下は予測ではなく、IEAの2030年データセンター電力需要約945TWhを基準にした**適用規模の感度試算**です。  
仮に、適用対象部分で2倍性能が成立し、同一処理量を約半分の実行エネルギーで処理できる場合を置きます。

| 全データセンター需要のうち適用可能な割合 | 2倍成立時の削減余地（仮定） | 電力単価 0.05 USD/kWh 換算 | 電力単価 0.10 USD/kWh 換算 |
|---:|---:|---:|---:|
| 0.1% | 約0.47 TWh/年 | 約2,360万USD/年 | 約4,730万USD/年 |
| 1% | 約4.73 TWh/年 | 約2.36億USD/年 | 約4.73億USD/年 |
| 5% | 約23.63 TWh/年 | 約11.8億USD/年 | 約23.6億USD/年 |
| 10% | 約47.25 TWh/年 | 約23.6億USD/年 | 約47.3億USD/年 |

この表が示すのは、非常に限定された適用率でも、成立した場合の経済価値が大きくなり得るということです。

しかも、この試算に含めていない価値があります。

- 追加サーバー購入の延期又は削減
- データセンター増設の延期
- 系統接続待ち容量の緩和
- 冷却設備・運用費用の削減余地
- 同一設備から得られる売上スループットの増加
- 端末側での性能向上又はバッテリー負担低下

## 8.4 比較対象：ハードウェア増設の投資規模

計算性能を物理供給力の増加で補う場合、投資は極めて大きい。

| 公表例 | 投資規模 |
|---|---:|
| TSMC：米国での先端半導体製造投資計画 | 合計 **1,650億USD** |
| Intel：Ohioの先端半導体工場2棟計画 | **280億USD超** |
| SEMI：2025年の世界の前工程ファブ装置投資予測 | **1,100億USD** |

本件は、これらを不要にするという主張ではありません。  
半導体製造能力そのものは必要です。

しかし、

> **数百億から千億ドル規模で物理供給力を増設する前に、  
> 既に存在する計算資源から引き出せる実効性能が大きく残っているなら、  
> それを検証しない理由がない。**

という投資判断を生みます。

# 9. 検証リソースと初期投資の小ささ

## 9.1 初期検証では新CPUもファブも不要

この仮説を最初に判定するために、必要なのは以下です。

- 市販又は既存のx86-64 / ARM計算機
- 低レイヤー命令生成又は変換を扱える技術者
- 性能カウンタ取得環境（Linux `perf`、PMU等）
- 代表的ベンチマーク
- 出力一致検証
- 実験用の機械語フォーマット／メモリ運用計画の試作層

初手で不要なもの：

- 半導体工場
- 新CPU製造
- 新ISAの業界標準化
- 大規模クラウド契約
- 顧客向け製品化
- 営業組織

## 9.2 段階別の必要リソース概算

以下は事業計画ではなく、検証に必要な工数の概算です。

| 段階 | 目的 | 人員目安 | 期間目安 | 必要資源 |
|---|---|---:|---:|---|
| Phase 0 | 既存実測の再現、hardware counter取得、比較条件の公平化 | 低レイヤー性能技術者 1–2名 | 1–3週間 | 既存PC／サーバー、Linux計測環境 |
| Phase 1 | メモリ運用命令化の最小モデル、四条件比較 | 2–3名 | 1–2か月 | コード生成・実行ハーネス、複数CPU環境 |
| Phase 2 | 新フォーマットの実験仕様、複数ワークロード検証 | 3–5名 | 2–6か月 | AI/DB/数値演算ベンチ、再現ログ |
| Phase 3 | 製品化又はハード支援判断 | 結果に依存 | 結果に依存 | 投資判断はここで初めて必要 |

最も重要なのは、**Phase 0～1で不成立なら早期に止められる**ことです。  
投資額が巨大化する前に、仮説の核を既存ハード上で判定できます。

## 9.3 投資リターンの非対称性

この仮説の投資構造は非対称です。

| 結果 | 損益 |
|---|---|
| 不成立 | 数週間から数か月の低層検証コストで停止できる |
| 一部成立 | 特定ワークロード向け最適化として価値が残る可能性がある |
| 2倍級成立 | 既存計算資源、データセンター容量、AI計算、端末性能へ横断的な価値を持ち得る |
| ハード支援まで成立 | CPU／アクセラレータ設計へ波及し、産業基盤技術になる可能性がある |

したがって、この技術は、

> **失敗時の初期損失は小さく、成立時の市場効果は計算基盤全体へ広がり得る。**

という検証対象です。

# 10. あなた側で最初に行うべきこと

私はこの件を日常的に監督するつもりはありません。  
あなた又はあなたが接続する適切な低レイヤー技術者が、まず次を行ってください。

## Phase 0：既存ログの再現と監査

- 行列積実測の再現
- 高度最適化コンパイラとの公平比較
- CPU、クロック、キャッシュ、メモリ帯域条件の記録
- `perf` / PMUによるcache miss、IPC、branch、memory bandwidth 等の取得
- 出力一致検証
- 実行順・warm-up・cache状態等の測定バイアス除去

判定：

| 判定 | 条件 |
|---|---|
| `PASS` | 性能差が再現し、差の主要機構を計測値で説明できる |
| `SUSPEND` | 性能差はあるが、比較条件又は原因分析が不足 |
| `FAIL` | 公平条件では性能差が消える、又は再現不能 |

## Phase 1：メモリ運用命令化の最小試作

最低限、以下のメモリ役割を明示し、実行計画へ落としてください。

| 役割 | 内容 |
|---|---|
| Read-only input | 読み取り専用入力 |
| Reused input | 反復中に再利用する対象 |
| Output target | 出力／書き込み先 |
| Temporary state | 一時状態 |
| Preparation target | 次段のために先行準備できる対象 |
| Movable target | 移送又は再配置対象 |
| Conflict target | 競合又は同期が必要な対象 |

## Phase 2：機械語フォーマットの実験モデル

アセンブラ文法ではなく、機械側へ渡したい構造単位を試作してください。

最低対象：

- operation block
- memory role
- dependency relation
- movement / preparation plan
- verification / stop condition

初期実装は、実験フォーマットを既存x86-64又はARM命令へ展開する層で構いません。  
目的は、フォーマットが保持する構造によって、従来形式より高い性能又は低い資源占有を再現可能に生むかを測ることです。

## Phase 3：四条件比較

| 条件 | 機械語／実行構造改善 | メモリ運用改善 | 判定目的 |
|---|---:|---:|---|
| Baseline | なし | なし | 元の基準 |
| Format-only | あり | なし | 機械語側の効果 |
| Memory-only | なし | あり | メモリ側の効果 |
| Combined | あり | あり | 統合効果 |

初期成立判定：

| 判定 | 条件 |
|---|---|
| `PASS` | 同一出力を保持し、適用可能な基準処理で `Combined / Baseline >= 2.0x` を再現性あるログとして確認 |
| `PARTIAL PASS` | 2倍未満でも、統合効果が個別改善を明確に上回る |
| `SUSPEND` | 測定条件、実装又は原因分析が不足 |
| `FAIL` | 追加効果がなく、原理仮説が支持されない |

# 12. 結論

この技術の根は、二つの前提を疑うことにあります。

## 第一の前提：ノイマン型の誤読

ノイマン型の本質は、

> 命令とデータを同じメモリへ置いたこと

だけではありません。

本質は、

> **制御・処理手順を保持対象へ変えたこと**

です。

その原理を正しく延長すれば、メモリの配置、移送、再利用、先行準備、並列運用そのものも保持・実行対象へ変えられます。

## 第二の前提：機械語フォーマットの固定化

現行の機械語フォーマットは、歴史的互換性と過去の設計制約の上に成立した形式です。  
高度なCPU最適化が存在しても、処理構造とメモリ運用を第一級の実行情報として保持する目的には最初から最適化されていません。

この形式を再設計し、メモリ運用側の制御と統合すれば、既存計算機の中に残る損失を大きく回収できる可能性があります。

既に、既存x86-64上の限定的な基準実験では、実行構造とアクセス順の差だけで平均1.691倍が観測されています。  
2倍へ到達するための追加差分は約18.3%です。

そして、この仮説の初期判定は、

- 新CPUなし
- 新工場なし
- 既存ハード
- 少人数の低層実装者
- 数週間から数か月の検証

で開始できます。

一方で成立した場合、その価値は、急増するデータセンター計算需要、AI計算、科学技術計算、データ処理、端末性能へ横断的に及び得ます。

> **初期投資は小さく、成立時の射程は計算基盤全体に及ぶ。  
> だから、検証できる人間がすぐに走るべき技術です。**

私はこの件を抱えて止まるつもりはありません。  
あなたが価値を認識し、適切な実装者へ接続できるなら、進めてください。
